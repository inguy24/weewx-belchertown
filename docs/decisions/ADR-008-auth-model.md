# ADR-008: Auth model for API and realtime endpoints

## Context

Downstream of [ADR-037](ADR-037-inbound-traffic-architecture.md): all public traffic goes through one reverse proxy; api and realtime bind to loopback (or trusted-LAN in cross-host) and aren't internet-reachable; upstream calls originate server-side.

**End-user authentication is not in scope and not provided.** Weewx skins (Belchertown, Seasons, Smartphone, Weather Eye, Beautiful Dashboard) don't implement user logins, and we don't either. A weather site is public information; users who need access control add it at the proxy layer (Apache basic-auth, Authelia, Cloudflare Access, etc.) as a deployment choice.

The remaining narrow question: **when proxy and inner services run on different hosts, what stops a third party on the LAN from bypassing the proxy and hitting an inner service directly?**

Future privileged surfaces (configuration UI per [ADR-027](ADR-027-config-and-setup-wizard.md), update mechanism per [ADR-028](ADR-028-update-mechanism.md)) define their own auth in their own ADRs.

## Options considered

| Option | Verdict |
|---|---|
| A. None — trust the network | Rejected for cross-host deploys (anyone on LAN can bypass proxy); fine for single-host where loopback is the trust boundary. |
| B. Shared secret in a header | **Selected.** Tiny code surface, no certs to manage, optional (single-host doesn't need it), setup wizard auto-generates and distributes. |
| C. mTLS between proxy and inner service | Rejected — cert management operationally hostile; wildly disproportionate. |

## Decision

**Shared secret in `X-Clearskies-Proxy-Auth` header, used only when needed.**

- **Same-host deploys** (inner service bound to `127.0.0.1`/`::1`): no shared secret needed. Network isolation is the trust boundary. A local attacker who can connect to `127.0.0.1:8765` already has shell access and can read the DB file directly.
- **Cross-host deploys** (inner service bound non-loopback): shared secret strongly recommended. Proxy injects the header on every request; inner services compare against the configured value. If bound non-loopback with the secret unset, the service **starts anyway** but emits a loud startup warning every 60 s. We don't refuse to start — would block debugging / custom topologies.

The shared secret is generated by the configuration UI ([ADR-027](ADR-027-config-and-setup-wizard.md)) and stored in environment variables on each host. End user never sees it. Power users skipping the wizard generate one with `openssl rand -hex 32` (documented recipe).

User-facing auth: not provided. Stack repo's INSTALL.md includes a worked Apache basic-auth example and pointers to Authelia / Cloudflare Access.

## Consequences

- **No login code in any repo.** No user database, no session store, no password handling, no lockout, no password-reset. Zero CVE surface for authentication.
- **Browser SPA → API has no auth credentials at all.** Same-origin proxying handles everything. CORS is a non-issue. No "embed key in JS" footgun.
- **Same-host deploys see no auth config at all.** Install, run, done.
- **Cross-host deploys carry one extra env var per side** (the shared secret). Configuration UI handles it.
- **`/health` is on a separate loopback-bound port** per security baseline; the secret middleware doesn't apply to it.
- **Future privileged surfaces define their own auth.** Configuration UI, update mechanism, any admin UI.

### Trade-offs accepted
- **Shared secret is plaintext on the wire.** Fine on a typical home LAN; not fine on shared infrastructure (office LAN, multi-tenant cloud). Docs state the threat model: secret is for casual LAN trust, not adversarial network conditions. Operators in adversarial environments use single-host deploys, wireguard/tailscale, or mTLS (future ADR if demand emerges).
- **No revocation, no expiry on the secret.** Manual rotation: regenerate and redeploy on both hosts. Acceptable — secret is not user-facing and gates only read-only weather data. Configuration UI makes regeneration one-button.
- **Startup warning for non-loopback-without-secret can be ignored.** A determined operator who exposes the inner service directly will succeed. Operator's deployment, operator's call.

### Repos affected
- **api / realtime:** middleware that, when `WEEWX_CLEARSKIES_PROXY_SECRET` is set, requires the header on every request to the public listener. `/health` on a separate loopback listener — unaffected.
- **dashboard:** nothing. Never authenticates.
- **stack:** docs cover the optional secret, proxy-side header injection, reverse-proxy auth patterns.

## Implementation guidance

### Configuration

Same env var on api and realtime: `WEEWX_CLEARSKIES_PROXY_SECRET=<generated>`.

- **Set:** every request to the public listener must carry `X-Clearskies-Proxy-Auth: <secret>`. Constant-time compare; mismatch → 401.
- **Unset, loopback bind:** service starts normally, no auth enforced.
- **Unset, non-loopback bind:** service starts, logs loud `WARNING` at startup and every 60 s thereafter:

```
WARNING: clearskies-api is bound to a non-loopback address (10.0.0.5)
without WEEWX_CLEARSKIES_PROXY_SECRET set. Any host that can reach this
address can read this service directly, bypassing your reverse proxy.
See SECURITY.md for the recommended cross-host config.
```

### Middleware (FastAPI sketch)

```python
@app.middleware("http")
async def proxy_secret_check(request, call_next):
    expected = os.environ.get("WEEWX_CLEARSKIES_PROXY_SECRET", "").strip()
    if not expected:
        return await call_next(request)
    provided = request.headers.get("X-Clearskies-Proxy-Auth", "")
    if not hmac.compare_digest(provided, expected):
        return Response(status_code=401)
    return await call_next(request)
```

### Proxy-side header injection

**Apache:** `RequestHeader set X-Clearskies-Proxy-Auth "${CLEARSKIES_PROXY_SECRET}"`
**Caddy:** `reverse_proxy api-host:8765 { header_up X-Clearskies-Proxy-Auth {env.CLEARSKIES_PROXY_SECRET} }`
**nginx:** `proxy_set_header X-Clearskies-Proxy-Auth $clearskies_proxy_secret;`

### Documentation

Each component's `SECURITY.md` includes an "Auth model" section: states plainly no end-user auth (consistent with weewx skin ecosystem), describes the optional shared-secret mechanism, recommends reverse-proxy auth (Apache basic-auth as simplest example, Authelia/Cloudflare Access for richer needs), notes future privileged surfaces define their own auth.

Stack repo's INSTALL.md: one-paragraph "if you want to password-protect your site" with 5-line Apache basic-auth recipe + pointers to Authelia / Cloudflare Access.

## Out of scope
- OAuth, OIDC, JWT, sessions, password storage.
- mTLS between proxy and inner services.
- Per-user accounts, scope-limited tokens.
- Authentication of upstream provider calls (provider credentials per [ADR-006](ADR-006-compliance-model.md), held server-side).
- Auth for any privileged surface — covered by future ADRs as those surfaces appear.
