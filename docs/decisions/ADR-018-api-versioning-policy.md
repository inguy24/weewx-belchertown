# ADR-018: API versioning policy

## Context

clearskies-api is the OpenAPI-defined contract between dashboard and api. The plan locks URL-path versioning (`/api/v1/...`). This ADR locks **when** to bump the major version, what counts as breaking, the deprecation policy, and the error format.

## Decision

### URL-path versioning, major bumps only on breaking changes

Version lives in the URL: `/api/v1/...`. Bump to `/api/v2/...` ONLY when a change is breaking. Non-breaking changes ship within the existing version.

### Breaking-change checklist (requires major bump)

- Removes an endpoint, query parameter, or response field.
- Changes a field's type or nullability.
- Renames a field or endpoint.
- Changes default behavior (default sort, default page size, default unit, etc.).
- Tightens validation on a previously accepted request shape.

### Non-breaking changes (ship within current major)

- Adds a new endpoint, optional query param, optional request field, or response field.
- Loosens validation.
- Marks a field deprecated (kept in response with `deprecated: true` in OpenAPI; not removed until next major).

### Deprecation behavior (signal-only, no support promise)

The project does **not commit to a support window** for any API version — software is provided AS-IS under GPL v3 ([ADR-003](ADR-003-license.md)). Prior major versions may continue to function in a release after the next major ships, or may be removed without advance notice. Operators upgrade at their own cadence and risk.

When a release **does** keep older routes around, those routes **may** emit `Deprecation` and `Sunset` headers (RFC 8594) as a courtesy technical signal describing what subsequent releases plan to do. The headers are advisory, not a contractual commitment.

### Error format

All endpoints (every version) use **RFC 9457 Problem Details for HTTP APIs** (`application/problem+json`). Error format itself is non-versioned — consistent across `/api/v1`, `/api/v2`, etc.

## Options considered

| Option | Verdict |
|---|---|
| A. URL-path versioning + major bumps only on breaking changes (this ADR) | **Selected** — visible, cacheable, easy to reason about. |
| B. Header-based versioning (`Accept: application/vnd.clearskies.v1+json`) | Rejected — invisible in URLs; harder to debug and cache. |
| C. Query-string versioning (`?v=1`) | Rejected — bloats URLs; conflicts with caching semantics. |
| D. No versions / continuous evolution | Rejected — leaves no contract for pinned dashboards or third-party clients. |

## Consequences

- OpenAPI spec auto-generated by FastAPI; served at `/api/v1/openapi.json` + Swagger UI at `/api/v1/docs`.
- Dashboard code-generates a typed client from the OpenAPI spec at build; client pinned to a specific api version.
- Each major version lives in its own router module in clearskies-api source. v2 starts as a copy of v1 with the breaking changes applied.
- v1 → v2 bump is a Phase 5+ event; not anticipated for v0.x development.

## Out of scope

- gRPC / GraphQL alternative protocols.
- Per-endpoint experimental flags — Phase 6+ if needed.
- Formal RFC process for breaking changes — informal during v0.x; formalize when public users adopt.
