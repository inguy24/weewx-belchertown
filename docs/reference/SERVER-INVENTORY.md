## Hosts overview

| Host | Role | Network |
|---|---|---|
| **Ratbert** (LXD host) | LXD container host on home network | manages bridges `br-vlan2`, `br-vlan7`, `br-internal` |
| **weewx** (LXD container) | WeeWX 5.3.1 engine, generates static HTML, runs MQTT publisher | eth0 `192.168.2.121` (br-vlan2) |
| **cloud** (LXD container) | Apache HTTPS frontend + EMQX MQTT broker + Nextcloud | eth0 `192.168.7.2` (br-vlan7), eth1 `10.0.1.3` (br-internal) |

The two containers are **on different VLAN bridges**, so traffic between them traverses the home network's L3 router (Mikrotik).

## Static HTML site (the public weather site)

### Generation

- **Generated by:** weewx daemon on `weewx` container (WeeWX 5.3.1)
- **Generation path on weewx container:** `/var/www/weewx/`
- **Skin source:** `/etc/weewx/skins/Belchertown/`
- **Configured by:** `/etc/weewx/weewx.conf` `[StdReport][[Belchertown]]`, `HTML_ROOT = /var/www/weewx`

### Sync to public-facing host

**Mechanism: LXD shared disk device** (NOT rsync, NOT cron, NOT a weewx skin).

```
Ratbert host:   /mnt/weewx/   (single source of truth on the host)
                ├── bind-mounted into weewx  container at /var/www/weewx  (lxd device "weewx-output")
                └── bind-mounted into cloud  container at /var/www/weewx  (lxd device "weewx-web")
```

Both containers see the same files instantly. weewx writes; Apache on cloud reads.

### Public serving

- **Apache vhost:** `/etc/apache2/sites-enabled/weather-ssl.conf` on cloud
- **DocumentRoot:** `/var/www/weewx/`
- **TLS termination:** **Apache on cloud (port 443)** — Let's Encrypt cert at `/etc/letsencrypt/live/weather.shaneburkhardt.com/`
- **Public URL:** https://weather.shaneburkhardt.com
- **HTTP→HTTPS redirect:** vhost on port 80 redirects to https
- **HSTS:** enabled (`max-age=31536000`)

> **NOTE:** the ratbert-lxd reference doc says the reverse proxy is nginx-proxy-manager (NPM). For weather.shaneburkhardt.com, **NPM is not in the path** (or is operating in pure pass-through). TLS termination is on cloud's Apache. NPM's exact role for this hostname is **not yet verified**.

### Stale paths on cloud (NOT used by the live site)

- `/var/www/weather/` — owner `shane:shane`, last modified 2025-06-11. **Older site, abandoned.**
- `/var/www/bak1/` — backup from 2022. **Abandoned.**
- `/var/www/html/` — Nextcloud (cloud.shaneburkhardt.com), not weather-related.

## MQTT (real-time data stream to browser)

### Broker

- **Software:** EMQX (Erlang `beam.smp`)
- **Host:** `cloud` container
- **Ports:**
  - `1883` — MQTT over TCP (where weewx publishes)
  - `8083` — MQTT over WebSocket (where browsers connect via wss proxy)
  - `18083` — EMQX admin dashboard (LAN only)

### Publisher (weewx → broker)

- **Extension:** `matthewwall/weewx-mqtt` (`/etc/weewx/bin/user/mqtt.py`)
- **Enabled in:** `weewx.conf` line 635 — `restful_services = ..., user.mqtt.MQTT`
- **Config:** `weewx.conf` `[StdRESTful][[MQTT]]`
- **Configured publish URL:** `mgtt://weewx:<REDACTED>@cloud.shaneburkhardt.com:1883` (see CREDENTIALS.md)
- **Configured topic:** `weewx`
- **🚨 BUG: scheme is `mgtt://` instead of `mqtt://` — typo. weewx-mqtt cannot parse this and is failing silently.**

### Subscriber (browser → broker via wss)

- **Configured in:** `weewx.conf` `[StdReport][[Belchertown]][[[Extras]]]`
- **Browser connects to:** `wss://weather.shaneburkhardt.com:443/mqtt`
- **Topic:** `weewx/loop`
- **Auth:** username `weewx-web`, password in CREDENTIALS.md (stored plaintext in weewx.conf on server)

### TLS-termination + WS-upgrade chain

```
Browser
  │ wss://weather.shaneburkhardt.com:443/mqtt
  ▼
Apache vhost weather-ssl.conf on cloud
  │ TLS terminated; URL /mqtt with Upgrade: websocket header
  │ RewriteRule: ^/mqtt(.*) → ws://localhost:8083/mqtt$1 [P,L]
  ▼
EMQX listener on cloud port 8083 (MQTT-over-WS)
```

The proxy chain is **correctly wired in Apache**. The break is upstream: weewx isn't publishing because of the `mgtt://` typo, so subscribers receive nothing.

### Topic mismatch (potential second issue)

- weewx-mqtt publishes under base topic `weewx`
- Belchertown subscribes to `weewx/loop`

The matthewwall weewx-mqtt extension by default appends `/loop` for loop packets, so this likely resolves correctly. But this should be verified once the `mgtt://` typo is fixed.
